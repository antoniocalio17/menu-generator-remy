================================================================================
HEYRA — MENU GENERATOR  |  Architecture & Developer Notes
================================================================================

OVERVIEW
--------
Generates a weekly canteen lunch menu for two tracks (meat / vegetarian),
Monday to Friday, grounded in a real product catalogue (products.csv).

Pipeline:
  Catalogue → Planner (LLM) → fix_budget → Validator → Exporter → API → UI

Each step is a separate module under engine/. The pipeline is wired together
in api/main.py (FastAPI) and exposed through a browser UI in static/.

Run with:
  uvicorn api.main:app --reload    (from the menu-generator/ directory)
  Then open http://localhost:8000


================================================================================
MODULE: engine/catalogue.py
================================================================================

PURPOSE
  Loads products.csv and provides typed, filtered samples of products
  to be passed to the LLM as candidate ingredient pools.

DATA TYPES
  Product (TypedDict)
    One row from products.csv — product_id, product_name, ingredient_group,
    cost_per_100g_eur, energy_kcal_per_100g, dietary_class,
    allergen_gluten, allergen_nuts, allergen_dairy.

  IngredientInput (TypedDict)
    A (product_id, quantity_g) pair used when computing stats for a dish.

  MealStats (frozen dataclass)
    Result of compute_meal_stats: cost_eur and calories_kcal for a dish.

  TrackPool = dict[role, list[Product]]
    One track's candidate pools keyed by role name.

  WeeklyPools = dict[track, TrackPool]
    Both tracks together: {"meat": TrackPool, "vegetarian": TrackPool}.

ROLE GROUPS
  All 29 ingredient groups from the CSV are mapped to 5 roles:
    protein_meat  — meat, poultry, fish, seafood, charcuterie
    protein_veg   — legumes, eggs, plant proteins, cheeses, nuts
    carb          — grains, rices, pastas, breads
    vegetable     — vegetables, mushrooms
    sauce         — sauces, oils, herbs, spices, condiments, creams,
                    milks, yogurts, sweeteners, flours, seeds, fruits,
                    plant-based beverages

  Fruits, sweeteners, seeds etc. are kept in the sauce pool so the LLM
  can use them when culinarily appropriate (chutneys, nut roasts, etc.)
  rather than being arbitrarily excluded.

CLASS: Catalogue
  __init__(csv_path)
    Reads the CSV, keeps only is_available=1 rows.
    Builds two DataFrames:
      _df     — full filtered table, used for sampling/filtering
      _by_id  — same table indexed by product_id for O(1) lookup

  get_candidates_by_role(role, track, n=15, exclude_ids=None)
    Returns up to n randomly sampled Products for a given role and track.
    - Vegetarian track: filters to dietary_class in {vegan, vegetarian}
    - Meat track + protein_meat role: filters to dietary_class == "meat"
    - Raises ValueError on unknown role or track name.

  build_weekly_pools(n_per_role=15, exclude_ids=None)
    Calls get_candidates_by_role for all roles of each track and returns
    a WeeklyPools dict. Called once per generation run.

  get_product_by_id(product_id)
    O(1) lookup by product_id. Returns Product or None if not found /
    not available.

  get_cheaper_in_group(ingredient_group, max_cost, track)
    Returns all products in the same group costing less than max_cost,
    sorted cheapest first. Used by fix_budget in validator.py.

  compute_meal_stats(ingredients: list[IngredientInput])
    Sums cost and calories for a list of (product_id, quantity_g) pairs.
    Returns a MealStats(cost_eur, calories_kcal).
    Raises ValueError if any product_id is unknown.


================================================================================
MODULE: engine/output_format.py
================================================================================

PURPOSE
  Defines the JSON schema the LLM must return, validated by Pydantic v2.

MODELS
  Ingredient  — product_id (int), quantity_g (float, must be > 0)
  Dish        — dish_name (non-empty str), ingredients (non-empty list)
  TrackPlan   — one Dish per day: monday … friday
  WeeklyPlan  — week_start (date), meat (TrackPlan), vegetarian (TrackPlan)

  Track-first structure (not day-first) so the two tracks are independent
  and the LLM can reason about each track as a coherent unit.


================================================================================
MODULE: engine/groq_llama.py
================================================================================

PURPOSE
  Calls the Groq LLM (llama-3.3-70b-versatile via OpenAI-compatible API)
  with the candidate pools and returns a validated WeeklyPlan.

CONFIGURATION
  Model:        llama-3.3-70b-versatile
  Temperature:  0.2  (low — JSON reliability over creativity;
                variety comes from random pool sampling, not temperature)
  Response:     JSON mode (ResponseFormatJSONObject)
  Retries:      3 attempts max (_MAX_RETRIES)
  Backoff:      [2s, 5s, 10s] on Groq 429 rate-limit errors
  API key:      loaded from engine/.env (GROQ_API_KEY)

RETRY LOGIC
  The retry loop handles two classes of LLM failure:

  1. Schema errors (Pydantic ValidationError)
     The LLM returned JSON that doesn't match WeeklyPlan. The error
     is injected into the next prompt under "PREVIOUS ERRORS" so the
     model can self-correct.

  2. Invented product IDs (_check_product_ids guardrail)
     After a successful JSON parse, every product_id in the plan is
     checked against the catalogue. Any unknown ID is an error — the
     LLM hallucinated a product that wasn't in the candidate pools.
     Same feedback loop: errors go into the next prompt as PREVIOUS ERRORS.

  3. Rate limit (RateLimitError / HTTP 429)
     Groq rejects the request because we've exceeded their quota.
     The code waits 2s, 5s, then 10s before retrying silently.
     If all backoff slots are used, raises PlannerError to the caller.
     NOTE: a 429 does not count against _MAX_RETRIES.

  On exhausted retries: raises PlannerError with the last error and
  the last raw LLM response for debugging.

KEY FUNCTIONS
  _format_products(products)   — formats a product list for the prompt
  _format_track(pool)          — formats a full TrackPool for the prompt
  _prompt_messages(pools, week_start, errors)
                               — builds the system + user messages
  _check_product_ids(plan, catalogue)
                               — guardrail: returns errors for invented IDs
  generate(week_start, catalogue)
                               — main entry point, runs the full loop


================================================================================
MODULE: engine/validator.py
================================================================================

PURPOSE
  Two concerns: budget auto-fix and hard constraint validation.

CONSTANT
  WEEKLY_BUDGET_EUR = 20.10  (per track, per week)

fix_budget(plan, catalogue) -> WeeklyPlan
  Called before validate(). Silently adjusts the plan if over budget.
  Strategy: find the single most expensive ingredient across all dishes,
  swap it for the cheapest available product in the same ingredient_group,
  skip swaps that would duplicate another dish's ingredient set.
  Repeats until within budget or no cheaper swap is possible.

validate(plan, catalogue)
  Collects ALL constraint violations (does not fail fast) then raises
  PlanValidationError(errors: list[str]) if any are found.
  Checks:
    - All product_ids exist in catalogue
    - Meat track: every dish has at least one meat/poultry/fish/seafood/
      charcuterie ingredient
    - Vegetarian track: no ingredient with dietary_class == "meat"
    - Each track's weekly cost ≤ WEEKLY_BUDGET_EUR


================================================================================
MODULE: engine/exporter.py
================================================================================

PURPOSE
  Converts a validated WeeklyPlan into a rich WeekSummary with cost,
  calories, and allergen data. Renders to Markdown or a JSON-serializable
  dict for the API.

KEY TYPES
  IngredientDetail  — product_id, name, quantity_g, cost_eur,
                      calories_kcal, allergens
  DishDetail        — dish_name, ingredients, total_cost_eur,
                      total_calories_kcal, allergens
  TrackSummary      — dishes (dict[day, DishDetail]), weekly_cost_eur,
                      weekly_calories_kcal, allergens
  WeekSummary       — week_start, meat (TrackSummary), vegetarian (TrackSummary)

KEY FUNCTIONS
  build_summary(plan, catalogue) -> WeekSummary
  summary_to_dict(summary, week) -> dict   (used by the API)
  to_markdown(summary) -> str              (used by main.py CLI)

================================================================================
API: api/main.py  (FastAPI)
================================================================================

Routes all include {year} so menus across calendar years work correctly.

  POST /api/generate/{year}/{week}
    Runs the full pipeline (generate → fix_budget → validate → export).
    Saves result to data/menus/{year}_w{week:02d}.json.
    Returns the WeekSummary dict.

  GET  /api/menu/{year}/{week}
    Returns the stored WeekSummary for a year+week.
    404 if not yet generated.

  PUT  /api/menu/{year}/{week}/{track}/{day}
    Chef edits a single dish. Validates before saving.
    Body: { dish_name: str, ingredients: [{product_id, quantity_g}] }
    Returns the updated WeekSummary.

  GET  /api/catalogue
    Returns all available products (used by the UI ingredient search).

  GET  /
    Serves index.html (frontend).

  /static/*
    Serves static/ directory (CSS, JS).

================================================================================
FRONTEND: static/app.js
================================================================================

STATE
  currentYear  — ISO year currently displayed (int)
  currentWeek  — ISO week number currently displayed (int, 1–52/53)
  menuData     — last fetched WeekSummary from the API (or null)
  catalogue    — list of all products, loaded once on page init

WEEK/YEAR NAVIGATION
  getISOWeek(date)          — standard ISO week number from a JS Date
  getWeeksInYear(year)      — 52 or 53 depending on the year
  isoWeekMonday(year, week) — returns the Monday Date for a given week
  formatDateRange(year, week) — "Jun 23 – Jun 27" label for the header
  changeWeek(delta)         — increments/decrements currentWeek;
                              rolls currentYear over when crossing week 1
                              or the last week of the year

API CALLS
  loadCatalogue()           — GET /api/catalogue, stores in catalogue[]
  loadWeek(week, year)      — GET /api/menu/{year}/{week}
  generateWeek()            — POST /api/generate/{year}/{currentWeek}
  saveDish(track, day)      — PUT /api/menu/{year}/{week}/{track}/{day}

RENDERING
  renderCalendar(data)      — builds the 6-column CSS grid:
                              [track label] [Mon] [Tue] [Wed] [Thu] [Fri]
                              Day headers show the actual date (e.g. "Mon, Jun 23")
  renderTrackRow(...)       — renders one row (meat or vegetarian)
  buildDishCard(track,day,dish)
    Each card has two modes toggled by class:
      .dish-view   — read mode: name, cost/kcal, allergen pills,
                     "Details" button, "Edit" button
      .dish-edit   — edit mode: name input, ingredient rows with qty inputs,
                     "+ Add ingredient" (opens modal), Save / Cancel

MODALS
  Details modal (#details-overlay)
    Opened by "Details" button on any dish card.
    Shows a table: ingredient | grams | allergens
    Footer: total cost EUR · total calories kcal

  Add-ingredient modal (#modal-overlay)
    Opened by "+ Add ingredient" in edit mode.
    Search box filters the full catalogue list.
    Select a product + quantity, click "Add to dish" → appends an
    ingredient row to the active editor without saving to the server.
    Save only happens when the chef clicks "Save" on the dish card.
